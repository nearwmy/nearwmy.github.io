cropperjs 中预览跨域图片填坑记

Posted by near on 2020-03-16
#cropperjs #cors #no-cros #img
fontend

问题概述

前几天业务中发现了个不同系统内查看同一个图片资源有的会出现图片加载失败的情况，乍一看像是出现跨域问题了，其实不然。这问题涉及比较多细节，今天在这里整理一下。

强制缓存和协商缓存

强制缓存标识头：Cache-Control / Expires, Cache-Control
优先级比 Expires 高,如果命中了强缓存，status code 依然是 200，不过会标识 from xxx cache

协商缓存标识头：【Last-Modified，If-Modified-Since】/【ETag、If-None-Match】
如果命中协商缓存，status code 会返回 304 并标识 Not Modified

图片都是存储在 s3 上的，图片地址并没有做重定向处理，所以到了前端获取到的地址是一个第三方资源地址，根据 s3资源请求返回的响应头的缓存标识，cache-control 为 no-cache， 同时返回了 Last-Modified / Etag, 所以在强缓存被命中的情况下，不会再次向服务器发起请求。这是导致以上问题的原因之一

条件型 cors 响应和无条件型 cors 响应

条件型 cros:

  • 根据有无 origin 头，来确定是否返回 Access-Control-Allow-Origin
  • 根据不同的 origin，返回不同 Access-Control-Allow-Origin 值

无条件型 cros： 将 Access-Control-Allow-Origin 固定写死为 * ，允许任意网站访问

上面说到，因为是第三方资源请求，涉及到跨域请求，所以服务在响应的时候会返回 Access-Control-Allow-Origin 头部标识，值为 当前 origin，在没有 origin 的时候，则不返回此字段。但是如果内部不同系统（ origin 不同的情况下）去请求同一个资源地址时，因为资源地址相同，所以会命中强制缓存，这时就会出现缓存错乱的情况，因为缓存的 header 上 跨域头 Access-Control-Allow-Origin 并不是当 origin，这时控制台会报错图片加载失败。

这个问题其实官方文档 MDN-HTTP访问控制（CORS） 已经给出了解决方案：

在响应首部加上： Vary: Origin, 让同一个 url，在不同系统中有不同的缓存

以上方法可以解决加载图片的问题，但是这样做之后，依然有一个问题：上传图片的时候使用了 cropperjs 去裁切，编辑头像进行裁剪的时候当前图片依然加载失败的问题，倒不是因为图片跨域不能加载，而是 cropperjs 在解决图片跨域加载会命中缓存导致加载的失败的情况下，给图片 url 添加了 timestamp 参数，每次去重新请求，s3 服务是对所有参数做了校验的，所以单独加参数会导致验证失败。

这里可能会疑问不就是再次请求图片吗？为什么会加载失败？ 这里和服务端集成 s3，设计缓存策略有关，确实应该在 304 的时候依然返回跨域头，但是并没有，现在想通过其他的方式看能不能紧急解决这个问题。到这里其实问题已经很清楚了，就是没有在正确的请求中响应跨域头，最后我们只能在 nginx 这层将 Access-Control-Allow-Origin 重写为 *， 然后返回给前端来解决。

Http 首部出现重复字段怎么办

配置 nginx 期间，由于 s3 本身有下发 Access-Control-Allow-* 相关头部，nginx 只能 add_header 操作字段，所以出现了重复的字段，根据 HTTP RFC 2616规定，是可以出现多个重复字段的，不过并没有明确规定出现重复字段时浏览器应该如何处理，所以每个浏览器内部实现逻辑可能不一致，有的以第一次出现的为准，有的以最后出现的为准，chrome 上试了下，好像是以最后出现的字段值为准。
